# 项目级 AI 编码规范

## 适用范围
- 本规范适用于整个仓库，重点约束后端目录 `src/main/java` 下的 Java 代码。
- 后续 AI 或人工修改现有后端代码时，默认先阅读本文件，再开始编码或重构。

## 总体原则
- 所有新增或修改的注释必须使用中文。
- 优先沿用项目现有分层：`controller -> application -> persistence/entity/mapper -> infrastructure/config`。
- 在不改变业务行为的前提下，优先提升可读性、可维护性和注释完整度。
- 不要为了“看起来有注释”而写空话，注释应说明职责、边界、原因和关键实现步骤。

## 中文注释要求

### 类、接口、枚举、记录类型
- 每个类、接口、枚举、`record`、配置类、异常类、过滤器、消息组件都必须有中文类注释。
- 类注释至少说明：
  - 这个类型的职责是什么。
  - 它位于哪一层，通常被谁调用。
  - 如果存在明显的设计意图，也要顺手说明。

### 方法
- 所有 `public` 方法必须有中文方法注释。
- 复杂的 `protected` / `private` 方法也必须有中文方法注释。
- 方法注释至少说明：
  - 方法用途。
  - 关键输入输出含义。
  - 是否承担权限校验、状态流转、数据转换、消息投递等关键职责。
- 方法内部存在以下情况时，必须补充行内中文注释：
  - 有多个步骤的业务流程。
  - 状态机校验或权限分支。
  - 金额、库存、时间窗口等容易误解的计算。
  - 消息发送、重试、补偿、幂等处理。
  - 兼容逻辑、兜底逻辑、性能取舍。

### 常量、字段、记录组件
- 业务常量、配置常量、topic 名称、状态阈值、响应头名称等都必须说明作用。
- `record` 组件、重要实体字段、配置属性字段建议加中文注释，说明业务含义，而不是只写字段字面意思。
- 对于明显的框架字段，如 `createdAt`、`updatedAt`，也应说明它们由谁填充、何时更新。

### 禁止事项
- 不要写“给变量赋值”“调用方法”这种无信息量注释。
- 不要把英文文档注释直接混进后端主代码里，优先改成中文。
- 不要在简单 getter/setter 里堆砌重复废话；如果字段注释已经足够，可保持简洁。

## 后端分层约束

### Controller
- 负责接收请求、参数校验入口、调用应用服务、返回 `ApiResponse`。
- 控制器方法必须有中文注释说明接口用途。
- 复杂业务不要直接堆在控制器里，应下沉到应用服务。

### Application Service
- 承担核心业务编排、权限判断、状态流转校验、实体转换。
- 方法顶部要说明业务目标，方法内部要按步骤补中文注释。
- 如果存在“先查数据、再校验、再写库、最后发消息”的流程，必须把每一步写清楚。

### Entity / Mapper / DTO / VO / Event
- `Entity` 重点说明字段的业务意义和审计字段来源。
- `Mapper` 重点说明查询意图、保存语义、SQL 更新目的。
- `DTO / VO / Event / Response record` 重点说明每个字段或组件代表什么。

### Config / Infrastructure
- `@Bean` 方法必须说明这个 Bean 解决什么问题、会被谁使用。
- 安全、消息、对象存储、跨域、OpenAPI 等基础设施代码，既要说明“做什么”，也要说明“为什么这样配”。

## 修改现有代码时的额外要求
- 修改方法逻辑时，同步更新对应中文注释，避免代码和注释脱节。
- 如果发现旧注释与真实行为不一致，应优先修正注释或代码，不要保留错误说明。
- 新增业务状态、错误码、topic、配置项时，必须一起补注释。

## 验证要求
- 只要修改了后端 Java 代码，完成后至少执行一次后端编译校验。
- 默认验证命令：

```bash
mvn -q -DskipTests compile
```

## 输出风格建议
- 注释先写“为什么/做什么”，再写“怎么做”。
- 优先短句、直白、可搜索的表达，避免堆砌术语。
- 对维护者最有帮助的信息优先写，不必追求面面俱到，但不能缺关键上下文。
